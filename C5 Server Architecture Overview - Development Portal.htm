C5 Server Architecture Overview

This page is intended to provide a brief overview of the c5 server architecture/framework at the very top level i.e .from intercepting requests to executing the corresponding commands. Specifics of individual modules are covered elsewhere.

Contents

  1. 1 Background
  2. 2 High-Level Overview of Production Environment
  3. 3 Load Balancing and Session Management
    1. 3.1 Server Toggling
  4. 4 Core "Server" classes:
  5. 5 Server Configuration
    1. 5.1 Adding a new config setting to an existing config

Background


C5 is a completely custom application server architecture, roughly equivalent to a Servlet-based architecture such as tomcat. That is, the server:
  • listens on a socket for http requests from clients
  • starts a new thread to handle each incoming request (each request is an XML "command")
  • executes business logic based on the request/command to generate an appropriate xml response
  • returns the response to the client.

High-Level Overview of Production Environment


The diagram below shows a high-level picture of the travelfusion production server environment. The C5 servers themselves are firewall-ed off from public access. All external traffic comes through a hardware load balancer (either one physical machine or a cluster) which handles all requests to api.travelfusion.com.



Load Balancing and Session Management


On the live environment our overall server platform consists of multiple TripPlanner servers running on individual pcs, so some kind of load balancing is required to farm out requests efficiently across the servers.

The only user "sessions" which span more than one request-response exchange are the "Routing Sessions", the core search->booking request/response sequence which forms TravelFusions primary web service. Every search session is assigned a RoutingId a 16-character unique id which is included in every subsequent request and response in the sequence. The first character represents which server the session was started on, the rest are random.

Currently c5 does not support any kind of persistent session handling, all session info is kept in an in-memory timeout cache on each server. Thus if a user starts a routing session on one server then all subsequent requests in the sequence need to go to the same server. For this reason we have an additional level of request re-direction on top of the basic round-robin load balancing, to match the routingId contained in the xml request to the correct server.

On our live environment we use a hardware load balancer to perform basic round-robin balancing on all incoming requests. We then use c5 itself to handle the session-based redirection. This is handled by a separate ServerSocket listener, TpBalanceServer, which listens on a different port (8769) and forwards the requests to the 8760 port on the appropriate server, or to itself, if the request contains a routingId.


Server Toggling


Toggling in this case defines whether or not a server is included in the pool of active servers which new search requests can be load balanced too. Servers can be toggled off manually e.g. to perform an upgrade or if they are needed to perform cpu or database intensive tasks (such as the nightly cleanup processing). There is also an automated monitor application (a Servlet which polls all servers every 10 seconds) which will toggle off servers and perform a restart if they detect that a server is having problems (not responding to polls, memory load passes a threshold etc.).

Toggling is controlled via a set of database tables which contain the current "toggle" status of each server.


Core "Server" classes:

project.tripplanner.TripPlanner
This is the core singleton "manager" class which acts as the central point of access to all the shared datasources and modules used by the server.

project.tripplanner.server.TpCommandServer
The "server" itself, i.e. the class which listens on the actual ServerSocket (port 8760) and kicks off the threads to handle each incoming request.

project.tripplanner.TripPlannerMain
The c5 "executable" called from the startup shell scripts. This class initialises the TripPlanner singleton and all its datasources/modules including the TpCommandServer. It also includes the static methods for shutting down the server.

project.tripplanner.server.TpCommandHandler
The thread created by TpCommandServer to handle each request. It parses the xml, validates it, creates an instance of the corresponding "Command" class and passes execution to it to perform the business logic and populate the response, then writes the response back to the socket.

Server Configuration


C5 also uses a bespoke system for initialising various server properties from xml configuration files e.g. session timeouts, database settings, various constants etc. The system used can be thought of as somewhere between a simple .properties file and a more complex bean-instantiation system such as EJB or Spring context configs.

Configuration settings are mapped to corresponding java classes, postfixed with Config, e.g. the <Database> element will be mapped to DatabaseConfig.java. The children of the element then get mapped to the private static fields* of the class with the same name (via reflection). Static getters and setters of the class are then be used to access the property values at runtime. 

Further, C5 also supports nested config elements, where a child of a config element could also contain children. An example of nested configuration is <WebSpiderStage> contains <proxyManager> which in turn contains its own children. The java class, in this case ProxyManager, mapped to a nested config element must implement an interface ConfigClass.

Configuration files are kept in the c5/java/config. All config files must start with a <Config> element containing optional host and address attributes - the hostname and ip address of the server the config should be used for. At startup c5 looks for a config file matching the local host and loads the corresponding settings. The <import> flag can be used to import settings from another config file and thus provide a simple hierarchical structure for inheriting common values and overriding only server-specific variations.

In general all configs will end up inheriting from tripplanner.xml which specifies the default settings for all config classes.



* yes, this is bad Java practice, and ideally should be refactored at some point.


Adding a new config setting to an existing config

Instructions for adding a new configuration property foo to an existing config:
  • Add a new private static variable called foo to the java class.
  • Add a public static getter method, e.g. getFoo() which returns the new field with clear documentation stating what the value represents.
  • Add a <foo> child element to the parent config in tripplanner.xml with a suitable default

Adding a new config class

Instructions for adding an entirely new configuration class MySettings:
  • Create a new class MySettingsConfig.java with clear comments stating the purpose of the settings
  • Add a new top-level element <MySettings> to tripplanner.xml
  • Add all required fields to the MySettingConfig as described above
  • Add a reference to the config class to ConfigService.doInit(), i.e. addConfig(MySettingsConfig.class) (please also see the interface UseConfig for another way to add config class in a service)
